# Karpathy에서 영감을 받은 Claude Code 가이드라인

> 새 프로젝트 [Multica](https://github.com/multica-ai/multica)를 확인해 보세요 — 재사용 가능한 스킬을 갖춘 코딩 에이전트를 실행하고 관리하는 오픈소스 플랫폼입니다.
>
> X에서 팔로우하기: [https://x.com/jiayuan_jy](https://x.com/jiayuan_jy)

[Andrej Karpathy의 관찰](https://x.com/karpathy/status/2015883857489522876)에서 도출한 LLM 코딩 함정에 대한 단일 `CLAUDE.md` 파일로 Claude Code의 동작을 개선합니다.

[English](./README.md) | [简体中文](./README.zh.md) | 한국어

## 문제점

Andrej의 게시글에서:

> "모델은 당신을 대신해 잘못된 가정을 하고 확인도 없이 그대로 진행합니다. 혼란을 관리하지 않고, 명확한 설명을 요청하지 않고, 불일치를 드러내지 않고, 트레이드오프를 제시하지 않으며, 반박해야 할 때도 반박하지 않습니다."

> "코드와 API를 지나치게 복잡하게 만들고, 추상화를 남발하고, 죽은 코드를 정리하지 않습니다... 100줄이면 충분한 것을 1000줄짜리 비대한 구조로 구현합니다."

> "작업과 무관한 코드나 주석도 충분히 이해하지 못한 채 변경하거나 삭제하는 부작용이 여전히 발생합니다."

## 해결책

하나의 파일에 담긴 네 가지 원칙으로 이러한 문제를 직접 해결합니다:

| 원칙 | 해결하는 문제 |
|-----------|-----------|
| **코딩 전에 생각하기** | 잘못된 가정, 숨겨진 혼란, 누락된 트레이드오프 |
| **단순함 우선** | 과도한 복잡성, 비대한 추상화 |
| **정밀한 수정** | 무관한 편집, 건드리지 말아야 할 코드 수정 |
| **목표 중심 실행** | 테스트 우선, 검증 가능한 성공 기준 |

## 네 가지 원칙 상세 설명

### 1. 코딩 전에 생각하기

**가정하지 마세요. 혼란을 숨기지 마세요. 트레이드오프를 드러내세요.**

LLM은 종종 하나의 해석을 조용히 선택하고 실행합니다. 이 원칙은 명시적인 추론을 강제합니다:

- **가정을 명시적으로 서술** — 불확실하면 추측하지 말고 질문하기
- **여러 해석 제시** — 모호함이 있을 때 조용히 선택하지 않기
- **필요할 때 반박** — 더 단순한 접근법이 있다면 말하기
- **혼란스러울 때 멈추기** — 불명확한 부분을 지적하고 설명 요청하기

### 2. 단순함 우선

**문제를 해결하는 최소한의 코드. 추측성 코드 금지.**

과도한 엔지니어링 경향에 대응합니다:

- 요청 범위를 넘는 기능 추가 금지
- 일회성 코드에 대한 추상화 금지
- 요청되지 않은 "유연성"이나 "설정 가능성" 금지
- 불가능한 시나리오에 대한 에러 처리 금지
- 200줄을 50줄로 줄일 수 있다면 다시 작성하기

**검증 기준:** 시니어 엔지니어가 이게 과도하게 복잡하다고 말할까요? 그렇다면 단순화하세요.

### 3. 정밀한 수정

**반드시 필요한 부분만 수정하세요. 자신이 만든 문제만 정리하세요.**

기존 코드를 편집할 때:

- 인접한 코드, 주석, 포맷팅을 "개선"하지 않기
- 문제없는 것을 리팩토링하지 않기
- 기존 스타일에 맞추기, 다르게 하고 싶더라도
- 무관한 죽은 코드를 발견하면 언급만 하기 — 삭제하지 않기

변경으로 인해 고아 코드가 생겼을 때:

- 자신의 변경으로 인해 사용되지 않게 된 임포트/변수/함수 제거
- 기존에 존재하던 죽은 코드는 요청받지 않는 한 제거하지 않기

**검증 기준:** 변경된 모든 줄은 사용자의 요청에 직접 연결되어야 합니다.

### 4. 목표 중심 실행

**성공 기준을 정의하고. 검증될 때까지 반복하세요.**

명령형 작업을 검증 가능한 목표로 변환합니다:

| 이렇게 하지 말고... | 이렇게 변환... |
|--------------|-----------------|
| "유효성 검사 추가" | "잘못된 입력에 대한 테스트를 작성하고, 통과시키기" |
| "버그 수정" | "버그를 재현하는 테스트를 작성하고, 통과시키기" |
| "X 리팩토링" | "리팩토링 전후로 테스트가 통과하는지 확인" |

다단계 작업의 경우 간단한 계획을 서술합니다:

```
1. [단계] → 검증: [확인사항]
2. [단계] → 검증: [확인사항]
3. [단계] → 검증: [확인사항]
```

강력한 성공 기준은 LLM이 독립적으로 반복 실행할 수 있게 합니다. 약한 기준("작동하게 해줘")은 끊임없는 확인이 필요합니다.

## 설치

**옵션 A: Claude Code 플러그인 (권장)**

Claude Code 내에서 먼저 마켓플레이스를 추가합니다:
```
/plugin marketplace add forrestchang/andrej-karpathy-skills
```

그런 다음 플러그인을 설치합니다:
```
/plugin install andrej-karpathy-skills@karpathy-skills
```

이렇게 하면 가이드라인이 Claude Code 플러그인으로 설치되어 모든 프로젝트에서 사용할 수 있습니다.

**옵션 B: CLAUDE.md (프로젝트별)**

새 프로젝트:
```bash
curl -o CLAUDE.md https://raw.githubusercontent.com/forrestchang/andrej-karpathy-skills/main/CLAUDE.md
```

기존 프로젝트 (추가):
```bash
echo "" >> CLAUDE.md
curl https://raw.githubusercontent.com/forrestchang/andrej-karpathy-skills/main/CLAUDE.md >> CLAUDE.md
```

## Cursor에서 사용하기

이 저장소에는 커밋된 Cursor 프로젝트 규칙 ([`.cursor/rules/karpathy-guidelines.mdc`](.cursor/rules/karpathy-guidelines.mdc))이 포함되어 있어 Cursor에서 프로젝트를 열 때 동일한 가이드라인이 적용됩니다. 다른 프로젝트에서 규칙을 사용하는 방법과 Claude Code와의 관계에 대해서는 **[CURSOR.md](CURSOR.md)**를 참조하세요.

## 핵심 인사이트

Andrej의 말:

> "LLM은 특정 목표를 달성할 때까지 반복 실행하는 데 탁월합니다... 무엇을 하라고 지시하지 말고, 성공 기준을 주고 알아서 하게 하세요."

"목표 중심 실행" 원칙이 바로 이것을 포착합니다: 명령형 지시를 검증 루프가 있는 선언형 목표로 변환하는 것입니다.

## 효과 확인 방법

다음과 같은 상황이 보이면 가이드라인이 작동하고 있는 것입니다:

- **diff에서 불필요한 변경이 줄어듦** — 요청한 변경만 나타남
- **과도한 복잡성으로 인한 재작성이 줄어듦** — 코드가 처음부터 단순하게 작성됨
- **구현 전에 확인 질문이 옴** — 실수 후가 아닌
- **깔끔하고 최소한의 PR** — 불필요한 리팩토링이나 "개선" 없음

## 커스터마이징

이 가이드라인은 프로젝트별 지침과 병합할 수 있도록 설계되었습니다. 기존 `CLAUDE.md`에 추가하거나 새로 만드세요.

프로젝트별 규칙은 다음과 같이 섹션을 추가합니다:

```markdown
## 프로젝트별 가이드라인

- TypeScript strict 모드 사용
- 모든 API 엔드포인트에 테스트 필수
- `src/utils/errors.ts`의 기존 에러 처리 패턴 따르기
```

## 트레이드오프 참고

이 가이드라인은 **속도보다 신중함**에 비중을 둡니다. 사소한 작업(단순한 오타 수정, 명백한 한 줄 수정)의 경우 판단력을 발휘하세요 — 모든 변경에 완전한 엄격함이 필요한 것은 아닙니다.

목표는 중요한 작업에서 비용이 큰 실수를 줄이는 것이지, 단순한 작업을 느리게 만드는 것이 아닙니다.
